# Story 2.6: Brand Intelligence — Digital Asset Scraping & Analysis

## Story

**As a** operator (Fernando),
**I want** the system to automatically scrape and analyze a client's existing digital assets (or reference URLs),
**so that** the brand config is enriched with real data and the agents produce more accurate, brand-consistent output from day one.

---

## Status

Ready for Review

---

## Context

**Epic 2 — Sistema 0: Geração de Projeto (Briefing Pipeline)**

Dependência de 2.1 (briefing com URLs), 2.5 (surveys específicos com URLs de referência). Esta story foi adicionada ao PRD v0.3 (FR-27/28/29) pelo @architect após descoberta de que clientes têm ativos digitais que podem enriquecer automaticamente o brand config.

**Dois fluxos:**
1. Cliente com ativos existentes: URLs capturadas no survey inicial → scraping automático
2. Cliente sem ativos: URLs de referência nos surveys específicos → scraping como guia de estilo

## Acceptance Criteria

- [ ] **AC-1:** Survey inicial (GoHighLevel — Story 2.1) já inclui campos para URLs de ativos existentes (site, Instagram, Facebook, LinkedIn, landing pages). Verificar que os campos foram adicionados.

- [ ] **AC-2:** n8n workflow `brand-intelligence-scrape` que:
  - Triggered após recebimento do briefing (se URLs presentes) OU após surveys específicos (para URLs de referência)
  - Para cada URL: chama API de scraping (Apify ou Firecrawl) via HTTP Request node

- [ ] **AC-3:** Pipeline de extração por URL deve coletar:
  - (a) Paleta de cores dominantes (top 5 hex + percentual)
  - (b) Fontes detectadas (heading/body)
  - (c) Keywords recorrentes (top 20)
  - (d) Tom de voz inferido via análise Claude API (formal/casual/técnico)
  - (e) Screenshot da página (URL do screenshot salvo)
  - (f) Estrutura de conteúdo/navegação
  - (g) Links de redes sociais encontrados na página

- [ ] **AC-4:** Tabela `brand_intelligence` no Supabase:
  - `id` (uuid, PK)
  - `client_id` (FK → clients)
  - `source_url` (text, NOT NULL)
  - `source_type` (text, CHECK: 'website','instagram','facebook','linkedin','landing_page','reference')
  - `colors_detected` (jsonb — array de `{hex, percentage, role?}`)
  - `fonts_detected` (jsonb — `{heading?, body?}`)
  - `tone_detected` (text, nullable)
  - `keywords_extracted` (text[])
  - `screenshot_url` (text, nullable)
  - `content_structure` (jsonb)
  - `confidence_scores` (jsonb — score 0-1 por campo)
  - `raw_data` (jsonb)
  - `status` (text, CHECK: 'pending','scraping','analyzing','complete','failed')
  - `created_at` (timestamptz)

- [ ] **AC-5:** Fluxo com ativos existentes:
  - Briefing com URLs → scrape automático → brand_intelligence salvo → dados disponíveis para CMO na análise → brand_config pré-populado (Fernando confirma/ajusta)

- [ ] **AC-6:** Fluxo sem ativos (referências):
  - Surveys específicos com "URLs de referência" → mesmo pipeline → `source_type='reference'` → Agentes usam como guia de estilo (não como cópia)

- [ ] **AC-7:** Confidence score por campo: se score < 0.6 → campo marcado como `needs_manual_review` → Fernando notificado

- [ ] **AC-8:** Fallback: se scraping falhar para uma URL → registrar erro, continuar com demais URLs. Pipeline nunca bloqueado por falha de scraping.

- [ ] **AC-9:** `BrandConfigLoader` (Story 3.2) deve consultar `brand_intelligence` ao montar guardrails — dados de intelligence complementam (não substituem) brand_config manual

- [ ] **AC-10:** Teste: URL de site real no survey → scraping executa → brand_intelligence salvo → cores e tom de voz detectados

## Dev Notes

### BrandIntelligence Entity
[Source: architecture/copyzen-architecture.md#4.2]
```typescript
interface BrandIntelligence {
  id: string;
  client_id: string;
  source_url: string;
  source_type: 'website' | 'instagram' | 'facebook' | 'linkedin' | 'landing_page' | 'reference';
  colors_detected: Array<{ hex: string; percentage: number; role?: string }>;
  fonts_detected: { heading?: string; body?: string };
  tone_detected: 'formal' | 'casual' | 'technical' | null;
  keywords_extracted: string[];
  screenshot_url: string | null;
  content_structure: Record<string, unknown>;
  confidence_scores: Record<string, number>; // 0-1 per field
  raw_data: Record<string, unknown>;
  status: 'pending' | 'scraping' | 'analyzing' | 'complete' | 'failed';
  created_at: string;
}
```

### Scraping APIs
[Source: PRD#FR-27, architecture/copyzen-architecture.md#2.2]
- **Firecrawl:** REST API para scraping de websites. Retorna markdown + metadata. Boa para extração de texto e estrutura.
- **Apify:** Plataforma com Actors especializados. Melhor para redes sociais (Instagram Scraper Actor, etc.)
- Ambas acessíveis via HTTP Request node do n8n

### Análise de Tom de Voz via Claude
Prompt para CMO analisar o texto extraído e classificar tom de voz:
```
Dado este texto de um site/perfil de negócio, classifique o tom de voz como:
- formal: linguagem técnica, profissional, distante
- casual: coloquial, próximo, usa "você"
- technical: técnico mas não necessariamente formal, focado em dados/specs
Responda APENAS com uma das três opções. Texto: {extracted_text}
```
