# Story 1.2: Supabase Project & Initial Schema

## Story

**As a** developer (Fernando),
**I want** a Supabase project created with initial schema for clients and brand configs,
**so that** all systems have a place to persist client data with RLS isolation from day 1.

---

## Status

Approved

---

## Context

**Epic 1 — Foundation & Infrastructure Setup**

Dependência de 1.1 (monorepo com `packages/integrations/` scaffolded). O schema base do banco de dados é pré-requisito para todos os Epics 2-7.

**O que já existe:**
- `supabase/migrations/` com 5 arquivos SQL (criados em commits anteriores) — estes arquivos já cobrem parte do schema. Revisar e validar contra os AC desta story antes de criar migrations adicionais.

**O que deve ser validado/criado:**
- Supabase project criado na cloud (free tier)
- Tabelas `clients` e `brand_configs` conforme spec
- RLS habilitado
- Storage bucket `assets`
- Variáveis documentadas em `.env.example`

---

## Acceptance Criteria

- [ ] **AC-1:** Supabase project criado (free tier) com região `sa-east-1` (São Paulo) ou mais próxima disponível

- [ ] **AC-2:** Tabela `clients` criada com campos:
  - `id` (uuid, PK, default gen_random_uuid())
  - `name` (text, NOT NULL)
  - `email` (text, NOT NULL, UNIQUE)
  - `phone` (text)
  - `business_type` (text)
  - `status` (text, enum-like CHECK: 'active','inactive','onboarding', default 'onboarding')
  - `created_at` (timestamptz, default now())
  - `updated_at` (timestamptz, default now())

- [ ] **AC-3:** Tabela `brand_configs` criada com campos:
  - `id` (uuid, PK)
  - `client_id` (uuid, FK → clients.id ON DELETE CASCADE)
  - `primary_color` (text), `secondary_color` (text), `accent_color` (text), `background_color` (text), `text_color` (text)
  - `heading_font` (text), `body_font` (text)
  - `tone_of_voice` (text, CHECK: 'formal','casual','technical')
  - `custom_guidelines` (text, nullable)
  - `logo_url` (text, nullable)
  - `slogan` (text, nullable)
  - `keywords` (text[], default '{}')
  - `reference_images` (text[], default '{}')
  - `created_at` (timestamptz, default now())

- [ ] **AC-4:** RLS habilitado em ambas as tabelas com policies:
  - Service role key tem acesso total (para operações internas de agentes/n8n)
  - Acesso de usuário anon BLOQUEADO por padrão em ambas as tabelas

- [ ] **AC-5:** Supabase Storage bucket `assets` criado com acesso público para leitura de logos/imagens de referência

- [ ] **AC-6:** Migrations SQL versionadas em `supabase/migrations/` no monorepo (verificar se os arquivos existentes cobrem os AC acima; criar migration complementar se necessário)

- [ ] **AC-7:** Variáveis de ambiente documentadas/atualizadas em `.env.example`:
  ```
  SUPABASE_URL=
  SUPABASE_ANON_KEY=
  SUPABASE_SERVICE_ROLE_KEY=
  ```

- [ ] **AC-8:** Teste de integração em `tests/supabase/` valida:
  - Insert client → Insert brand_config com FK → Query retorna dados corretos
  - RLS bloqueia acesso anon direto às tabelas
  - Service role key tem acesso completo

## Dev Notes

### Tech Stack Reference
[Source: architecture/copyzen-architecture.md#3]
- Database: Supabase Cloud (PostgreSQL 15+)
- RLS: Row Level Security via service_role key para ops internas
- Storage: Supabase Storage para logos e imagens de referência

### Migrations Existentes
Os seguintes arquivos já existem em `supabase/migrations/` (criados em commit anterior). Verificar conteúdo antes de criar duplicatas:
- `20260314010000_extensions_and_functions.sql`
- `20260314020000_core_tables.sql`
- `20260314030000_pipeline_tables.sql`
- `20260314040000_leads_and_observability.sql`
- `20260314050000_rls_policies.sql`

### BrandConfig Interface Completa
[Source: architecture/copyzen-architecture.md#4.2]
A interface no banco deve corresponder exatamente à `BrandConfig` TypeScript definida no `packages/core`. Campos `background_color` e `text_color` são adições do @architect vs PRD original — incluir no schema.

### Multi-tenancy Pattern
[Source: architecture/copyzen-architecture.md#2.5]
Isolamento via RLS usando `client_id`. Agent server usa `SUPABASE_SERVICE_ROLE_KEY` que bypassa RLS — é o expected behavior para operações internas. Nunca expor service_role key em frontend.

```
supabase/migrations/          → migrations SQL (existente + novas)
tests/supabase/               → testes de integração do schema
.env.example                  → variáveis Supabase
```
