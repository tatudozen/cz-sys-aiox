# Story 2.5: Surveys Específicos & Geração do Plano de Projeto

## Story

**As a** operator (Fernando),
**I want** system-specific surveys to be sent after approval and a detailed project plan generated,
**so that** each system has the detailed input it needs to produce quality output.

---

## Status

Approved

---

## Context

**Epic 2 — Sistema 0: Geração de Projeto (Briefing Pipeline)**

Dependência de 2.4 (aprovação da proposta). Após aprovação, o sistema envia surveys específicos para cada sistema contratado (Conteúdo, FunWheel, Página de Vendas), coleta as respostas e usa o CMO para gerar o plano de projeto detalhado.

---

## Acceptance Criteria

- [ ] **AC-1:** Surveys específicos no GoHighLevel para cada sistema:
  - **Survey de Conteúdo (Sistema 1):** temas de posts, estilo visual preferido, concorrentes, exemplos de posts que gosta, frequência desejada
  - **Survey de FunWheel (Sistema 2):** narrativa de transformação, lead magnet topic, community VIP description, goal de indicações
  - **Survey de Página de Vendas (Sistema 3):** oferta principal, preço, depoimentos existentes, garantia, FAQ frequentes
  - Cada survey inclui campo "URLs de referência/inspiração" para Brand Intelligence (FR-28)

- [ ] **AC-2:** n8n workflow `dispatch-specific-surveys`:
  - Triggered pela aprovação da proposta (Story 2.4)
  - Input: `briefing_id`, `package`
  - Com base no pacote, envia APENAS os surveys dos sistemas contratados via email + WhatsApp

- [ ] **AC-3:** Tabela `system_surveys` no Supabase:
  - `id` (uuid, PK)
  - `briefing_id` (FK → briefings)
  - `system` (text, CHECK: 'content','funwheel','sales_page')
  - `responses` (jsonb)
  - `status` (text, CHECK: 'pending','received','complete')
  - `created_at` (timestamptz)

- [ ] **AC-4:** Webhook por survey que salva respostas em `system_surveys` com status `received`

- [ ] **AC-5:** Quando todos os surveys do pacote estão com status `received`, n8n chama CMO para gerar plano de projeto detalhado

- [ ] **AC-6:** Tabela `project_plans` no Supabase:
  - `id` (uuid, PK)
  - `briefing_id` (FK → briefings)
  - `client_id` (FK → clients)
  - `package` (text)
  - `systems` (jsonb — array com `{system, tasks[], estimated_days}`)
  - `status` (text, CHECK: 'draft','active','completed')
  - `created_at` (timestamptz)

- [ ] **AC-7:** Fernando notificado quando plano está pronto: "Plano de projeto para {client_name} gerado. Revisar antes de aprovar."

- [ ] **AC-8:** Teste end-to-end: aprovação → surveys enviados → cliente responde surveys → plano de projeto gerado → Fernando notificado

## Dev Notes

### Dispatch Logic por Pacote
[Source: PRD#FR-04]
```
IA → Survey de Conteúdo
A-R-T → Survey de FunWheel
E → Survey de Página de Vendas
Combo Leads → Survey de Conteúdo + Survey de FunWheel
Combo Cash → Survey de Conteúdo + Survey de FunWheel + Survey de Página de Vendas
```

### Prompt CMO para Project Plan
Contexto para o CMO: briefing completo + respostas de todos os surveys + pacote aprovado → Output: JSON com systems[], cada system com tasks[] ordenadas e estimated_days.

### URLs de Referência para Brand Intelligence
[Source: PRD#FR-28]
Os surveys devem ter campo "URLs de referência/inspiração" (sites ou perfis de concorrentes). Quando preenchido, aciona o mesmo pipeline de Brand Intelligence (Story 2.6) com `source_type='reference'`.
