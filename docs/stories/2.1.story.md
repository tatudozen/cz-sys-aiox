# Story 2.1: GoHighLevel Survey & Webhook Configuration

## Story

**As a** operator (Fernando),
**I want** the GoHighLevel initial survey connected to n8n via webhook,
**so that** client briefings arrive automatically in the processing pipeline.

---

## Status

Approved

---

## Context

**Epic 2 — Sistema 0: Geração de Projeto (Briefing Pipeline)**

Dependência de Epic 1 (Supabase schema operacional, n8n rodando na KVM4). Este é o ponto de entrada do pipeline — o briefing inicial do cliente.

**Infraestrutura existente:**
- GoHighLevel ativo com survey inicial parcialmente configurado
- n8n em queue mode na KVM4 (domínio n8n já definido)
- Supabase com tabelas base (migrations do Epic 1)

---

## Acceptance Criteria

- [ ] **AC-1:** Survey inicial no GoHighLevel configurado com campos:
  - Nome do negócio (obrigatório)
  - Segmento/nicho (obrigatório)
  - Público-alvo (obrigatório)
  - Objetivo principal (obrigatório)
  - Orçamento estimado (opcional)
  - Canal preferido de comunicação: WhatsApp/Email (obrigatório)
  - URLs de redes sociais existentes (Instagram, Facebook, LinkedIn, site) — opcional, para Brand Intelligence (FR-27)

- [ ] **AC-2:** Webhook configurado no GoHighLevel disparando para endpoint n8n: `https://{n8n-domain}/webhook/briefing-intake`

- [ ] **AC-3:** n8n workflow `briefing-intake` que:
  - Recebe payload do GoHighLevel
  - Valida campos obrigatórios
  - Salva raw data na tabela `briefings` no Supabase com status `received`

- [ ] **AC-4:** Tabela `briefings` criada no Supabase (verificar se já existe nas migrations; criar se necessário):
  - `id` (uuid, PK)
  - `client_id` (uuid, FK nullable → clients.id)
  - `raw_data` (jsonb, NOT NULL)
  - `source` (text, CHECK: 'ghl_survey','whatsapp')
  - `status` (text, CHECK: 'received','processing','analyzed','approved','needs_info', default 'received')
  - `created_at` (timestamptz, default now())
  - RLS habilitado (mesmo padrão de clients)

- [ ] **AC-5:** RLS habilitado na tabela `briefings`

- [ ] **AC-6:** Teste end-to-end manual: preencher survey → webhook dispara → n8n processa → registro no Supabase com status `received`

- [ ] **AC-7:** Tratamento de erro no n8n: se processamento falhar, registra em error log e envia alerta (n8n notification node ou email)

## Dev Notes

### Briefing Data Flow
[Source: architecture/copyzen-architecture.md#2.4]
```
GoHighLevel Survey → webhook → n8n workflow briefing-intake
  → Supabase: INSERT briefings (status=received)
  → Trigger: workflow cmo-analyze-briefing (Story 2.3)
```

### n8n Workflow Export
Exportar workflows n8n como JSON em `docs/n8n/workflows/` para versionamento no git. n8n não tem integração nativa com git — exportação manual após cada mudança significativa.

### GoHighLevel Survey Fields para Brand Intelligence
[Source: PRD#FR-27]
O campo "URLs de redes sociais existentes" é usado pelo workflow de Brand Intelligence (Story 2.6). Garantir que o survey capture essas URLs corretamente no payload JSON.

### Tabela briefings — Verificar Migration Existente
O arquivo `20260314030000_pipeline_tables.sql` provavelmente já contém a tabela `briefings`. Verificar antes de criar migration nova.

### File Locations
```
supabase/migrations/                → verificar/complementar
docs/n8n/workflows/briefing-intake.json  → criar (export n8n)
```
